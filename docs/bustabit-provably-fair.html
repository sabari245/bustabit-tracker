bustabit provably-fair: hash chain + crash point

Status: re-derived 2026-06-27 against bustabit's OWN verifier source (authoritative): github.com/bustabit/verifier — files src/constants.ts, src/utils/math.ts, src/utils/vx.ts, src/verifier.worker.ts. All constants/algorithm below are copied from that source. [confidence: high — this is the canonical impl]

Critical: there are THREE eras on two hash chains, each with a different HMAC key and a different chain-walk encoding. Using the wrong era's salt/encoding silently produces a plausible-but-wrong bust. This was the original bug in this repo.

1. Hash chain

  • Each game has a server seed = a 64-hex SHA-256 value ("game hash").
  • Chain is precommitted and consumed in REVERSE. prevGameHash = SHA256(currentGameHash). i.e. hashing game N's hash yields game N-1's hash.
  • So: game #1 used the LAST-generated hash; the first generated hash (terminating/inception hash) is the last game played in that chain.
  • Chain-walk encoding differs by chain (verifier.worker.ts):
    • PREVIOUS chain (v2, games 1..10,000,000): SHA256 of the hex STRING — sha256(bytesToHex(hash)).
    • CURRENT chain (games > 10,000,000): SHA256 of the RAW 32 bytes — sha256(hash).
    Walking a current-chain hash with hex-string encoding (or vice-versa) will never reach the terminating hash.
  • Verify a hash belongs to game offset k from the end: SHA256 it k times (correct encoding) → must equal the published terminating hash for that chain.

3. HMAC keys per era — THIS is where impls go wrong

The crash formula (§4) is identical across eras; only the HMAC key (and, for Vx, how it's derived) changes. The HMAC message is ALWAYS the raw 32-byte game hash (NOT the hex string).

Note: PREV_GAME_SALT / GAME_SALT are passed to HMAC as the 64-char hex STRING's UTF-8 bytes (the string is the key, not hex-decoded). Only the MESSAGE is hex-decoded to raw bytes. (math.ts: hmac(sha256, key, gameHash) where key=salt string and gameHash=hexToBytes(hashHex).)

The Vx salt from the 3rd seeding event — 000000000000000000011f6e135efe67d7463dfe7bb955663ef88b1243b2deea (block 831500, VX_GAME_SALT) — is NOT an HMAC key. It is the clientSeed fed into the BLS signing step (§5). Do not use it directly as a key.

4. Crash point from hash — CANONICAL algorithm (math.ts, verbatim)

House edge (~1%) is baked into the constant 99. There is NO separate divisible-by-101 instant-bust check — that older MoneyPot/v1 variant is NOT what bustabit runs.

function gameResult(key, gameHash) {   // key = salt-string-bytes or BLS signature; gameHash = raw 32 bytes
  const nBits = 52;
  const hash = bytesToHex(hmac(sha256, key, gameHash));  // HMAC-SHA256(key, msg=raw hash bytes)
  const r = parseInt(hash.slice(0, nBits / 4), 16);      // first 13 hex chars = top 52 bits
  let X = r / Math.pow(2, nBits);                         // 2^52 = 4503599627370496; X in [0,1)
  X = 99 / (1 - X);                                       // edge in the 99; (1-X) avoids div-by-zero
  return Math.max(1, Math.floor(X) / 100);               // floor to 2 decimals, min 1.00x
}

Constants summary: HMAC-SHA256; key=era salt/signature; msg=raw 32-byte hash; 52-bit mask 2^52=4503599627370496; first 13 hex chars; formula 99/(1-X); floor; min 1.00. ~1% of games floor below 1.00 and clamp to 1.00 (that IS the instant-bust / edge).

4b. Common WRONG variants (do NOT use)

  • divisible-by-101 + 100-numerator: if(divisible(hash,101))return 1.00; floor((100*2^52 - h)/(2^52 - h))/100. This is the old MoneyPot/bustabit-v1 algorithm. Gives results 0.01 higher than §4 and a literal 1.00 spike. bustabit does NOT use it.
  • message = hex string: feeding the 64-char hex string to HMAC instead of the 32 raw bytes. Wrong digest entirely.
  • wrong-era salt: using v2 salt for a current-chain game (or vice-versa), or using VX_GAME_SALT as an HMAC key. Wrong digest entirely.

5. Vx era (BLS) — games 10,000,001 .. 12,279,450 (vx.ts)

During the Vx era the HMAC key is a per-game BLS12-381 signature, not a static salt. Reproducible because the private key was published after the era ended.

// constants.ts
VX_GAME_SALT = "000000000000000000011f6e135efe67d7463dfe7bb955663ef88b1243b2deea"  // clientSeed (block 831500)
VX_PUB_KEY   = "b40c94495f6e6e73619aeb54ec2fc84c5333f7a88ace82923946fc5b6c8635b08f9130888dd96e1749a1d5aab00020e4"
VX_PRIV_KEY  = "1f0373e57723f4b862f2b9bd1279c9c6dcea73469fce6dcab9524b609c9c60f4"
VX_LAST_GAME = 12279450

// vx.ts: derive the HMAC key (a BLS signature) for a Vx-era game
function getVxSignature(clientSeed /*=VX_GAME_SALT*/, gameHash /*raw bytes*/) {
  const message = concatBytes(sha256(gameHash), utf8ToBytes(clientSeed));   // sha256(rawHash) || clientSeed
  const hashed  = bls12_381.longSignatures.hash(message);
  const signature = bls12_381.longSignatures.sign(hashed, hexToBytes(VX_PRIV_KEY)).toBytes(true);
  return signature;   // pass as `key` to gameResult(§4)
}

Library: @noble/curves bls12-381 longSignatures (min-pubkey-size variant). Rust equiv would need a bls12-381 crate; only required to verify the ~2.28M Vx-era games.

6. Worked verification recipe

  1. Determine the game NUMBER (decides era + chain-walk encoding).
  2. Pick key per §3: v2→PREV_GAME_SALT; Vx→getVxSignature(§5); classic→GAME_SALT.
  3. gameResult(key, hexToBytes(hash)) (§4) → should equal published bust.
  4. Chain walk with the correct encoding (§1) k times → terminating hash for that chain (§2).

Reference checked: hash df5d54b8ca42cb40ea80e8ba74b3281f7b9d49bc27fecbe104847de1e91e4929 (current classic era) → GAME_SALT key + raw-bytes msg + §4 → 38.65x (matches site). Same hash via the old v2-salt/hex-string/divisible path → 16.67x (wrong).

8. Cautions

  • Game number is required to pick the era — a bare hash is ambiguous. If only a hash is known, default to current classic (GAME_SALT); it will be wrong for v2/Vx games.
  • HMAC message is RAW bytes; salt key is the hex STRING's UTF-8 bytes. Don't mix these up.
  • Vx era needs BLS12-381; without it, games 10,000,001..12,279,450 can't be reproduced.
  • Re-derive one live game end-to-end after any change.
